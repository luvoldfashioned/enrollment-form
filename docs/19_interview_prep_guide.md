# 🎤 채용 과제 면접 가이드 (Q&A)

평가 기준표(Rubric)에 명시된 핵심 면접 연계 질문 5가지에 대해, 현재 제출된 프로젝트의 아키텍처에 맞춘 가장 논리적이고 시니어다운 모범 답안 가이드입니다. 면접 전 반드시 숙지하시길 권장합니다.

---

### Q1. 폼 상태 관리를 위해 어떤 방식을 선택했나요? 다른 방법과 비교했을 때 Trade-off는?

**A.** `React Hook Form(RHF)`과 `Zod` 조합을 선택했으며, 스텝 간 상태 공유를 위해 RHF의 `FormProvider(Context)`를 활용했습니다.

* **다른 방법과의 비교 (Trade-off)**:
  * **전역 상태 (Redux/Zustand)**: 폼의 입력값은 UI 상태에 가깝습니다. 이를 전역 상태로 관리하면 불필요한 보일러플레이트가 증가하고, 입력할 때마다 전역 스토어가 업데이트되어 불필요한 렌더링 병목이 발생할 수 있습니다.
  * **Formik 또는 순수 useState**: 제어 컴포넌트(Controlled) 방식이므로 타이핑마다 렌더링이 일어납니다. 반면 RHF는 **비제어 컴포넌트(Uncontrolled)** 방식을 기반으로 하여 렌더링을 최소화하고 성능을 극대화할 수 있습니다.
* 결론적으로 렌더링 최적화(성능)와 `Zod discriminatedUnion`을 통한 강력한 타입 추론의 이점을 얻기 위해 RHF를 선택했습니다.

### Q2. 단체 신청에서 참가자 10명의 이름/이메일을 입력하는 UX가 불편할 수 있는데, 개선한다면 어떻게 하겠나요?

**A.** 현재는 `useFieldArray`를 활용해 동적으로 인풋을 추가/제거하는 방식을 구현했지만, 실무에서 10명을 모바일이나 좁은 화면에서 일일이 타이핑하는 것은 피로도가 매우 높습니다. 이를 개선하기 위해 두 가지 기능을 제안합니다.

1. **엑셀/CSV 일괄 업로드 기능**: `.csv` 파일을 업로드하면 브라우저 사이드(PapaParse 등 활용)에서 파싱하여 폼 필드에 일괄 주입하는 기능.
2. **붙여넣기 파싱 (Paste to Fill)**: 엑셀 스프레드시트나 메모장에서 긁어온 텍스트(Tab 또는 콤마 구분자)를 인풋 창에 붙여넣었을 때, `onPaste` 이벤트를 가로채어 자동으로 참가자 명단 배열을 생성해주는 UX를 도입하겠습니다.

### Q3. 3단계에서 제출 실패 시 사용자에게 어떤 경험을 제공하나요? 네트워크 에러와 비즈니스 에러를 구분하나요?

**A.** 사용자 경험을 해치지 않기 위해 **데이터 보존**과 **에러의 명확한 분리**를 원칙으로 삼았습니다.

1. **데이터 보존 및 중복 방지**: 제출 실패 시 이전 폼 상태가 그대로 유지되며(새로고침 없이), 제출 버튼의 `isSubmitting` 속성을 통해 연타로 인한 중복 요청을 원천 차단했습니다.
2. **에러의 분리**:
   * **비즈니스 에러 (409 Conflict 등)**: `App.tsx`의 `try-catch` 문에서 `response.ok`가 아닐 때 서버가 던진 `message`를 그대로 파싱합니다. (예: "정원이 가득 차 수강을 신청할 수 없습니다", "이 강의에 동일한 이메일 주소로 신청된 내역이 이미 존재합니다.") 이를 통해 사용자가 무엇을 잘못했는지 구체적인 UI 경고 배너로 띄워줍니다.
   * **네트워크 에러**: `err instanceof Error`에 잡히지 않거나 서버 응답 자체가 없는 경우, "네트워크 오류가 발생했습니다. 잠시 후 다시 시도해 주세요"라는 범용 메시지를 띄워 일시적인 장애 상황임을 인지시킵니다.

### Q4. 유효성 검증을 클라이언트와 서버 양쪽에서 해야 하는 이유는 무엇인가요?

**A.** 두 검증은 목적이 다르고 상호 보완적이기 때문입니다.

* **클라이언트 검증 (UX 최적화)**: 사용자가 입력하는 즉시(혹은 스텝을 넘길 때) 잘못된 포맷(이메일 규칙, 인원수 제한 등)을 피드백하여 네트워크 비용과 사용자의 대기 시간을 절약합니다.
* **서버 검증 (데이터 무결성 및 동시성 방어)**: 클라이언트 코드는 악의적인 사용자에 의해 조작(Bypass/Postman 직배송)될 수 있습니다. 또한 수강 신청처럼 **동시성 문제(Race Condition)**가 발생할 수 있는 비즈니스 로직(예: 동시에 두 명이 남은 1자리에 신청)은 클라이언트가 방어할 수 없으므로, 데이터 무결성을 지키기 위해 최종 방어선인 서버 검증은 필수불가결합니다.

### Q5. 현재 3단계인 폼이 5단계로 확장된다면 현재 구조에서 어디를 바꾸겠나요?

**A.** 현재는 `App.tsx` 내에서 `step === 1`, `step === 2` 방식으로 하드코딩된 조건부 렌더링을 하고 있습니다. 5단계 이상으로 늘어난다면 유지보수가 매우 힘들어집니다.

1. **퍼널(Funnel) 패턴 도입**: 스텝을 관리하는 로직을 배열(Array) 형태의 설정 파일이나 `Funnel` 컴포넌트로 추상화하여, 단계의 추가/삭제가 `components` 배열 하나로 관리되도록 아키텍처를 변경하겠습니다.
2. **React Router 결합 (URL 상태 동기화)**: 단계가 길어질수록 중간에 새로고침 시 데이터가 날아가거나 1단계로 튕기는 경험이 치명적입니다. `/enroll/step1`, `/enroll/step2` 처럼 라우터 기반으로 단계를 분리하고, 전역 상태(Context)나 `localStorage`(현재 구현된 Auto-save)와 결합하여 사용자가 URL로 접근해도 해당 단계를 자연스럽게 이어나갈 수 있도록 구조를 개편하겠습니다.
