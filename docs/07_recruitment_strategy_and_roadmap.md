## 2. 🛠️ 높은 숙련도 획득을 위한 4대 고도화 개발 태스크

평가 기준표의 '높은 숙련도' 요건을 채우기 위해 로컬 코드에 즉시 구현할 고도화 상세 설계입니다.

### 📍 [태스크 1] 개인/단체 전환 시 확인 대화상자 (Confirm Dialog) 적용

* **요구사항**: 신청 유형(개인 ↔ 단체) 라디오 변경 시, 사용자가 기존에 작성하던 내용이 지워지지 않도록 **"신청 유형을 변경하면 작성 중이던 정보가 초기화됩니다. 변경하시겠습니까?"** 라는 확인 창을 띄웁니다.
* **구현**: React 컴포넌트 내에 커스텀 모달이나 브라우저 기본 `confirm` 창을 활용하여 사용자가 승인할 때만 상태 초기화 및 탭 전환을 수행하도록 핸들러를 수정합니다.

### 📍 [태스크 2] 에러 발생 시 에러 필드로 자동 포커스 및 스크롤 처리

* **요구사항**: 2단계에서 '다음 단계'를 누르거나 3단계에서 '제출'을 눌렀을 때 유효성 검사에 실패하면, **가장 상위에 있는 에러 필드로 부드럽게 스크롤을 이동하고 자동으로 포커스(focus)**를 줍니다.
* **구현**: React Hook Form의 `formState.errors`를 조회하여 첫 번째 에러가 발생한 필드의 `name`을 찾아내고, 해당 HTML Input 엘리먼트에 포커스를 지정합니다.

### 📍 [태스크 3] Zod `z.discriminatedUnion`을 사용한 판별 유니온 타입 분리

* **요구사항**: TypeScript와 Zod 스키마 연동 시 `any`나 느슨한 옵셔널 타입을 배제하고, `type` 속성을 판별자(discriminator)로 두어 개인/단체의 타입을 완전 분리합니다.
* **구현**:
  * `personalEnrollmentSchema`와 `groupEnrollmentSchema`를 분리 정의합니다.
  * 최상위에서 `z.discriminatedUnion('enrollmentType', [personalEnrollmentSchema, groupEnrollmentSchema])`로 결합하여 컴파일러가 타입 세이프하게 추론하도록 리팩토링합니다.

### 📍 [태스크 4] 선택 구현(추가 점수): 임시 저장 및 이탈 방지

* **요구사항**: 새로고침 시 데이터 복구(localStorage) 및 페이지 이탈 시 확인 대화상자 표시.
* **구현**:
  * `useEffect`를 활용해 폼 데이터가 변경될 때마다 `localStorage`에 draft를 저장합니다.
  * 브라우저 새로고침이나 창 닫기 시 `beforeunload` 이벤트를 걸어 이탈 방지 경고를 제공합니다.

## 3. 💬 면접 연계 질문 대비 모범 답안 가이드

이 과제의 핵심은 **"AI가 짜준 코드를 내가 완벽히 제어하고 있는가"**입니다. 면접관이 질문할 과제 연계 핵심 질문들에 대한 모범 답변 가이드라인입니다.

#### Q1. 폼 상태 관리를 위해 어떤 방식을 선택했나요? 다른 방법과 비교했을 때 trade-off는?

* **답변**: "복잡한 다단계 폼 상태의 정합성과 실시간 유효성 검증을 위해 **React Hook Form**과 **Zod** 조합을 선택했습니다.
  * *비교*: React Context나 Redux 등으로 직접 상태를 다룰 경우, 매 입력마다 전체 폼 컴포넌트가 불필요하게 리렌더링되어 성능 저하가 발생할 수 있습니다.
  * *장점*: React Hook Form은 비제어 컴포넌트 방식을 활용해 렌더링 성능을 극대화하면서도, `FormProvider`를 통해 다단계 스텝 컴포넌트들 간에 정합성 있는 상태 공유가 가능했습니다."

#### Q2. 단체 신청에서 참가자 10명의 이름/이메일을 입력하는 UX가 불편할 수 있는데, 개선한다면 어떻게 하겠나요?

* **답변**: "참가자 10명을 화면에서 하나하나 타이핑하는 것은 불편하므로, 실무 관점에서는 **CSV 파일 일괄 업로드(Excel Import) 기능**을 추가로 제공하겠습니다. 사용자가 정해진 템플릿 양식에 맞추어 이름과 이메일을 작성해 업로드하면, 파싱하여 폼 내의 `participants` 배열에 한 번에 바인딩(setValue)해 주는 방식을 설계하겠습니다."

#### Q3. 3단계에서 제출 실패 시 사용자에게 어떤 경험을 제공하나요? 네트워크 에러와 비즈니스 에러를 구분하나요?

* **답변**: "제출에 실패해도 기존에 입력된 데이터는 상태 저장소(`App.tsx` 내의 폼 컨텍스트)에 안전하게 보존되므로, 사용자는 처음부터 다시 작성할 필요 없이 즉시 재시도할 수 있습니다.
  또한, 일반적인 **네트워크 에러(500, 연결 실패)**는 단순 안내 팝업을 띄우고, **비즈니스 에러(정원 마감: COURSE_FULL, 중복 가입: DUPLICATE_ENROLLMENT)**는 에러 배너를 통해 사용자에게 구체적인 실패 원인을 인지시켜 다른 행동(다른 강의 선택 등)을 유도하도록 정교하게 구분했습니다."

#### Q4. 유효성 검증을 클라이언트와 서버 양쪽에서 해야 하는 이유는 무엇인가요?

* **답변**: "사용자 경험(UX)과 서비스의 안정성/보안(Security)은 서로 다른 레이어에서 지켜져야 하기 때문입니다.
  * **클라이언트 검증**: 사용자가 올바르지 않은 데이터를 입력했을 때 즉각 피드백을 제공하여 불필요한 네트워크 리소스를 낭비하지 않고 전환율을 높이기 위해 수행합니다.
  * **서버 검증**: 클라이언트 코드는 개발자 도구나 악의적인 API 직접 호출 등을 통해 손쉽게 우회할 수 있습니다. 따라서 비즈니스 무결성과 데이터베이스 안정성을 최종적으로 확보하기 위해 서버 측 유효성 검증은 필수적입니다."

#### Q5. 현재 3단계인 폼이 5단계로 확장된다면 현재 구조에서 어디를 바꾸겠나요?

* **답변**: "현재 단계 제어 및 부분 검증 구조가 이미 스텝 배열 형태로 유연하게 설계되어 있어, 스텝 확장 시 영향도가 매우 낮습니다.
  * `App.tsx`의 하위 스텝 컴포넌트 분기 처리 부분에 추가 스텝 컴포넌트(`Step4...`, `Step5...`)를 끼워 넣고, `totalSteps`를 5로 변경합니다.
  * `handleNextStep` 검증 제어부의 `fieldsToValidate` 배열에 신규 스텝의 필드 이름들을 명시해 주는 방식으로 확장에 유연하게 대처할 수 있습니다."
