## 1. 아키텍처 의사결정 (Architecture Decision) 관련 질문

### Q1. React Hook Form과 Zod를 결합하여 사용한 결정적 이유는 무엇인가요?

* **핵심 질문 의도**: 라이브러리를 무비판적으로 가져다 썼는지, 혹은 프론트엔드의 성능적 이슈와 개발 생산성 간의 트레이드오프를 인지하고 있는지 검증하기 위함입니다.
* **추천 모범 답변**:
  > *"가장 큰 이유는 **불필요한 리렌더링 억제를 통한 폼 성능 최적화**와 **타입 시스템의 일관성 유지**입니다.*
  > *리액트에서 단순 `useState`로 다단계 폼 전체를 관리하면, 사용자가 키보드를 칠 때마다 폼 전체와 하위 컴포넌트가 매번 리렌더링되는 성능 문제가 있습니다.*
  > *이를 방지하기 위해 비제어 컴포넌트 방식으로 렌더링을 억제하는 **React Hook Form**을 기본 뼈대로 잡았습니다.*
  > *그리고 유효성 검사 규칙은 **Zod 스키마**로 분리하여 한 곳에서 관리했고, Zod의 `z.infer`를 활용해 폼 데이터의 TypeScript 타입을 자동으로 추론하게 함으로써 검증 규칙과 타입 선언의 싱크가 맞지 않아 발생할 수 있는 휴먼 에러를 방지했습니다."*

### Q2. 수강 신청이라는 비즈니스 규칙(개인/단체 분기, 인원수와 참가자 매칭 등)을 Zod 스키마에 어떻게 녹여냈나요?

* **핵심 질문 의도**: 단순한 정규식 수준을 넘어, 조건부 입력에 대한 다이내믹 검증과 상호 연관된 필드 간의 교차 검증(Cross-Field Validation)을 다루는 문제 해결 능력을 봅니다.
* **추천 모범 답변**:
  > *"Zod의 **`superRefine`** 메서드를 활용하여 설계했습니다.*
  > *1단계에서 선택한 `enrollmentType`이 `'group'`(단체)일 때만 단체명, 대표 담당자, 담당자 연락처, 신청 인원수 필드가 필수값으로 작동하도록 조건부 검증을 수동으로 분기 처리했습니다.*
  > *특히, **사용자가 입력한 '신청 인원수(headCount)' 수치와 실제 하단 참가자 명단(participants) 배열의 길이가 정확히 정합하는가**에 대한 정밀 교차 검증도 이 `superRefine` 내부에서 일괄 캡처하여 화면상에 정확한 필드 경로(`path: ['group', 'participants']`)로 에러 메시지를 노출시켰습니다."*

### Q3. 백엔드 API가 완성되지 않은 상태에서, API 통신 레이어의 신뢰성은 어떻게 입증했나요?

* **핵심 질문 의도**: "프론트/백에 갇히지 않고 전체 제품을 책임지는 프로덕트 엔지니어"로서 API 연동과 협업을 주도하는 능력이 있는지 평가합니다.
* **추천 모범 답변**:
  > *"백엔드 개발 환경이 구성되지 않아도 브라우저 환경에서 실제 서버가 존재하는 것처럼 시뮬레이션하기 위해 **MSW(Mock Service Worker)**를 선행 도입했습니다.*
  > *개발 서버 실행 시 브라우저의 Service Worker 레벨에서 요청을 가로채도록 설정하여, 개발자 도구의 Network 탭에서도 실시간 통신 흐름이 모니터링되게 구성했습니다.*
  > *이를 통해 강의 목록 조회(`GET /api/courses`) 시 카테고리 쿼리 스트링에 반응하는 백엔드 필터링 동작을 시뮬레이션하고, 신청서 제출(`POST /api/enrollments`) 시 1.5초의 서버 지연을 주어 프론트엔드 단에서 더블 클릭 등에 의해 발생하는 **중복 전송(Double Submit) 현상을 방지하는 로딩 화면 UI가 정상 동작하는지 선제 검증**할 수 있었습니다."*

## 2. 상태 관리 및 UI/UX 디테일 관련 질문

### Q4. 여러 단계로 나뉜 폼에서 단계 이동(이전 단계/다음 단계) 시 기존에 사용자가 입력한 데이터가 날아가지 않는 비결은 무엇인가요?

* **핵심 질문 의도**: 멀티스텝 폼 구현 시 주니어들이 가장 흔하게 겪는 상태 초기화 버그를 어떻게 극복했는지 아키텍처적 상태 설계 방식을 물어봅니다.
* **추천 모범 답변**:
  > *"폼의 모든 원천 상태를 최상위 부모 컴포넌트인 `src/App.tsx`의 `useForm` 인스턴스에 일원화하여 소유하게 했습니다.*
  > *그리고 하위 단계 컴포넌트들을 **`<FormProvider>`** Context로 감싸서, 각 단계 컴포넌트가 화면(DOM)에서 마운트 해제되더라도 입력 상태 데이터는 부모 컨텍스트 메모리에 온전히 보전되도록 구축했습니다.*
  > *이를 통해 3단계에서 요약을 확인하다가 `[수정]` 버튼을 클릭해 1단계나 2단계로 직접 점프(Step Back)해도 사용자가 기존에 선택하거나 입력한 정보가 흔적 없이 유지되는 부드러운 사용성을 제공했습니다."*

### Q5. 2단계 단체 신청 시, 사용자가 인원수를 조절할 때 하단 참가자 인풋이 동적으로 실시간 증감하는 흐름을 어떻게 설계했나요?

* **핵심 질문 의도**: 가변 배열 형태의 폼 필드를 성능 저하 없이 다이내믹하게 컨트롤할 수 있는지 묻습니다.
* **추천 모범 답변**:
  > *"React Hook Form의 **`useFieldArray`** 훅과 **`useEffect`** 감시 패턴을 결합하여 해결했습니다.*
  > *사용자가 인풋 창에 값을 타이핑하거나 마우스 휠, 혹은 키보드 방향키 등으로 인원수(`group.headCount`)를 바꿀 때마다 이 변화를 `watch()`하여 `useEffect` 내에서 반응적으로 명단 크기를 대조합니다.*
  > *만약 입력값이 기존 명단 수보다 크면 그 차이만큼 빈 참가자 객체(`{ name: '', email: '' }`)를 **`append()`**하고, 인원이 줄어들면 명단의 끝부분부터 **`remove()`**하게 루프를 구성했습니다.*
  > *이 방식은 사용자가 입력 방식에 구애받지 않고 단일 상태의 변경에만 뷰가 안전하게 동기화되게 만들며, 동시에 기존에 작성해 둔 앞번호 참가자들의 입력 데이터는 그대로 보존되는 실무적인 사용성을 띱니다."*

## 💡 면접관을 매료시키는 실무 요약 팁

면접관이 **"가장 어려웠던 점이나 중점적으로 신경 쓴 부분은 무엇인가요?"**라고 질문한다면 다음과 같은 요약 카드로 쐐기를 박으세요.

> *"다단계 폼을 설계할 때 가장 경계한 것은 **'데이터의 파편화'**와 **'어설픈 에러 처리'**였습니다.*
> *이를 위해 최상단에 상태를 통합하고, 각 단계를 넘어갈 때는 전체 폼을 검사하지 않고**`trigger(['해당스텝필드'])`**를 호출하여 현재 단계의 검증을 통과해야만 다음 스텝의 진입을 허용하는 타이트한 분기 제어에 많은 공을 들였습니다.*
> *또한 단체 신청에서 개인 신청으로 신청 구분을 바꿀 때, 기존 단체 정보 필드의 잔여 데이터를 명시적으로 **`setValue('group', undefined)`** 처리하여 백엔드 검사 및 Zod 제출 스펙과 완전히 일치하도록 정밀하게 폼 세정(Clean-up) 작업을 연동한 부분이 기술적으로 가장 신경 쓴 세부 디테일입니다."*
